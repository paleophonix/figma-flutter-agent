# Записка к тестовому заданию
## Figma → Flutter Adaptive Layout Agent

*Эта записка сопровождает выполненное тестовое задание и честно обозначает три вещи: **фактический объём и класс задачи**, **метод**, которым она решена (vibe coding — оркестрация AI-агента), и **реальные границы** результата. Все метрики проверяемы прямым подсчётом по репозиторию.*

---

## Краткое резюме

- **Выданное ТЗ — это спецификация продукта** (25 разделов: от Figma-коннектора до инкрементальной синхронизации и IDE/CI-интеграций), а не объём одного тестового задания.
- **Метод решения — vibe coding.** Я выступал постановщиком, архитектором процесса и контролёром качества; код генерировал AI-агент под моим управлением. Для вакансии, где задача — *строить AI-агентов для автоматизации*, это не оговорка, а **прямая демонстрация целевого навыка**.
- **Результат — работающее решение с продакшн-обвязкой:** ~65 000 строк кода и **1 866 автотестов**, покрыто практически всё ТЗ; есть observability LLM-вызовов, тестовый контур на моках и контроль стоимости.
- **Класс задачи — компиляторный R&D** (трансляция дизайна в код через строгое промежуточное представление и AST). Метод оркестрации с ним справился.
- **Честно о границах:** приоритет отдан pixel-fidelity мобильного фрейма, а не multi-breakpoint responsive; геометрический слой — в активной доводке.

> **Вывод:** формат «тестовое задание на middle-позицию» не отражает ни объём поставленной задачи, ни метод, которым она решена.

Техническое задание содержит **25 разделов** и описывает целостный продукт: Figma Connector (REST / Dev Mode / Styles / Components), Design Tree Parser, Auto Layout Analyzer, Responsive Engine, Widget Generator, Component Detection, Design System Generator, Asset Pipeline, Navigation, Accessibility, Incremental Sync, Developer Preservation System, IDE- и CI/CD-интеграции, варианты компонентов и design tokens.

> В индустрии спецификация такого объёма соответствует продуктовому роадмапу для команды на месяцы.

## 2. Метод: как это сделано (vibe coding)

Решение построено методом **vibe coding**: я не пишу алгоритмы вручную — я формулирую задачу, архитектуру и инварианты, агент генерирует код, я проверяю результат и направляю на доработку. Цикл повторяется до зелёного состояния.

Ключевой момент: **vibe coding без дисциплины даёт неподдерживаемую кашу.** Продукт с 1 866 тестами, CI и нативным AST-слоем получается только при выстроенном процессе оркестрации. Что я выстроил:

- **Свод правил оркестрации** (`.claude/prompts/`): *anti-patching law* (запрет точечных костылей — только универсальные алгоритмы, работающие на любом макете), guardrails кодогенерации, FSM-протокол состояний, пошаговая дисциплина пайплайна.
- **Контроль галлюцинаций модели:** автоматические repair-петли по результатам `flutter analyze`; мутации сгенерированного кода идут через **настоящий Dart AST (Analyzer), а не регулярные выражения**; реестр системных дефектов модели вместо разовых заплаток.
- **Роль человека:** держать карту системы, проектировать процесс, задавать инварианты и принимать/отклонять вывод агента по объективным гейтам (тесты, статический анализ).

> Это и есть навык, который ищет вакансия: не «написать функцию», а **построить и удержать управляемый агентный конвейер, дающий промышленный результат**.

## 4. Инженерная обвязка (эксплуатационная зрелость)

Это не разовый генератор «по кнопке», а система, которую можно наблюдать, тестировать и держать в бюджете:

**Observability LLM-вызовов**
- Трейсинг в **PostHog** как `$ai_generation`, корреляция по `run_id`, span-имена под каждую фазу (generate / repair / refine).
- Стадии пайплайна логируются со структурным контекстом и `duration_ms` — видно, где время и где сбой.

**Тестовый контур (для недетерминированного LLM)**
- 1 866 тестов; **121 файл с моками LLM через `respx`** и offline-фикстурами — недетерминированный вывод проверяется детерминированно.
- Golden-генерация и **acceptance-тесты прямо на критерии приёмки ТЗ** (раздел 23).
- Детерминированный fallback без модели — для offline-CI и аварийного режима.

**Стоимость и стабильность**
- **Offline-first batch:** один раз выкачивается Figma-файл, дальше генерация идёт офлайн без квоты.
- Сжатие payload к модели, управление reasoning-токенами под разных провайдеров, retry на 429/5xx, кэш AST-реконсиляции.

## 6. Инженерный класс задачи

Это не «обёртка над Figma API». Реализован полноценный **транслятор (компилятор) дизайна в код**:

- **Строгое промежуточное представление (IR)** как контракт между этапами пайплайна.
- **Вычислительная геометрия:** каскад аффинных матриц преобразований, проверка геометрических инвариантов, безопасный fallback на растеризацию при сложных трансформациях.
- **Нативный AST-слой на Dart:** программные правки кода — через компилятор-парсер, а не регулярки.
- **Многоступенчатый агентный пайплайн:** генерация по строгой JSON-схеме → авто-ремонт по `flutter analyze` → опциональная визуальная доводка по пиксель-диффу.
- **Валидация как проверка инвариантов** (а не «не упало — значит работает»).

По классу это **R&D-уровень**, верхние ~10 % задач отрасли по сложности.

## 7. Честные границы результата

Сознательно выкладываю это сам — чтобы оценка была точной:

- **Responsive слабее, чем pixel-fidelity.** ТЗ просило mobile/tablet/desktop/web. Я **сознательно сместил фокус** в pixel-perfect compile-time для мобильного фрейма (разделы 24–25 ТЗ — моё уточнение продуктовой модели). Каркас адаптивности есть (breakpoints, `MediaQuery`/`LayoutBuilder`), но multi-breakpoint reflow — не главная сила решения. Это осознанный trade-off, а не недосмотр.
- **Геометрический слой — в активной доводке.** Аффинный планировщик включён по умолчанию недавно; на произвольном Figma-дереве возможны артефакты раскладки. Это R&D-ветка.
- **Алгоритмическое нутро** (геометрия, механика AST) реализовано агентом. Я отвечаю за замысел, инварианты, контроль и проверку — не за построчную реализацию. Это прямое следствие метода и я его не маскирую.
- **Git-история переинициализирована** и не отражает ход разработки (работа шла плотными итерациями с агентом).
