# Технический отчет по исследованию и проектированию: Figma → Flutter Adaptive Layout Agent (MVP)

## Секция 1: Figma REST API (2025–2026)

### Результаты исследования (факты и даты)

Для проектирования легковесного консольного интерфейса (CLI) без использования среды исполнения плагинов Figma критически важно разграничить возможности, доступные через удаленный HTTP REST API, и функции внутреннего Plugin API. Работа в headless-режиме накладывает жесткие рамки на выбор конечных точек запросов. Основное архитектурное решение заключается в выборе между полной загрузкой дерева файла через `GET /v1/files/:key` и точечным запросом конкретных ветвей через `GET /v1/files/:key/nodes?ids=...`.

При обработке крупных файлов с множеством страниц запрос всего документа приводит к избыточному потреблению оперативной памяти, росту сетевого трафика и частым ошибкам превышения лимитов памяти на стороне сервера Figma. Для MVP-решения, обрабатывающего целевой экран, наиболее эффективен запрос конкретных узлов `GET /v1/files/:key/nodes?ids=...`.

Параметр `depth` (глубина вложенности) при точечном запросе позволяет ограничить объем возвращаемого поддерева, что полезно для предварительного анализа структуры без загрузки содержимого глубоко вложенных слоев. Параметр `geometry=paths` заставляет API возвращать точные векторные пути (инструкции отрисовки). Для компиляции стандартных структур разметки этот параметр избыточен и не должен использоваться, так как он увеличивает размер JSON-ответа в десятки раз, замедляя разбор структуры.

Парсинг URL-адреса фрейма Figma на стороне CLI должен надежно извлекать уникальный идентификатор файла (`file_key`) и идентификатор целевого узла (`node-id`). В адресе браузера идентификатор узла кодируется с использованием дефисов (например, `123-456`), тогда как REST API Figma принимает только формат с двоеточием (`123:456`). CLI должен выполнять препроцессинг строки перед отправкой HTTP-запроса.

Параметры Auto Layout в JSON-дереве Figma напрямую отображаются на свойства виджетов Flutter, обеспечивая перенос адаптивной верстки.

|**Свойство Figma JSON**|**Возможные значения**|**Эквивалент во Flutter**|**Механизм трансляции в кодогенераторе**|
|---|---|---|---|
|`layoutMode`|`HORIZONTAL`, `VERTICAL`, `NONE`|`Row`, `Column`, `Stack`|Задает направление главной оси контейнера Flex или переключает в режим наложения.|
|`primaryAxisAlignItems`|`MIN`, `MAX`, `CENTER`, `SPACE_BETWEEN`|`MainAxisAlignment`|Соответствует значениям `start`, `end`, `center`, `spaceBetween`.|
|`counterAxisAlignItems`|`MIN`, `MAX`, `CENTER`, `BASELINE`, `STRETCH`|`CrossAxisAlignment`|Соответствует значениям `start`, `end`, `center`, `baseline`, `stretch`.|
|`layoutSizingHorizontal`|`FIXED`, `HUG`, `FILL`|`SizedBox`, `Expanded`, `IntrinsicWidth`|`FILL` внутри Row маппится в `Expanded`, `FIXED` — в `SizedBox(width:...)`.|
|`layoutSizingVertical`|`FIXED`, `HUG`, `FILL`|`SizedBox`, `Expanded`, `IntrinsicHeight`|`FILL` внутри Column маппится в `Expanded`, `FIXED` — в `SizedBox(height:...)`.|
|`layoutGrow`|`0`, `1`|`Expanded` / `Flexible`|При значении `1` оборачивает элемент в Flex-контейнер для заполнения пространства.|
|`layoutAlign`|`MIN`, `MAX`, `CENTER`, `STRETCH`|`Align` / `CrossAxisAlignment`|Значение `STRETCH` заставляет элемент растягиваться по поперечной оси родителя.|
|`layoutWrap`|`NO_WRAP`, `WRAP`|`Wrap`|При `WRAP` Flex-контейнер заменяется на виджет `Wrap` с автоматическим переносом строк.|
|`itemSpacing`|Числовой float|`SizedBox` или `spacing` в `Wrap`|Задает фиксированные интервалы между дочерними элементами верстки.|
|`paddingLeft/Right/Top/Bottom`|Числовой float|`Padding(padding: EdgeInsets.only(...))`|Оборачивает внутреннее дерево Flex-контейнера в отступы.|
|`strokesIncludedInLayout`|`true`, `false`|Поправка размеров в `BoxDecoration`|Определяет, включается ли толщина рамки (stroke) в расчетные границы элементов.|
|`constraints.horizontal/vertical`|`MIN`, `MAX`, `CENTER`, `STRETCH`, `SCALE`|`Align` / `Positioned`|Применяется только при `layoutMode: NONE` для позиционирования в `Stack`.|
|`layoutPositioning`|`AUTO`, `ABSOLUTE`|`Positioned` внутри `Stack`|Элементы со значением `ABSOLUTE` исключаются из потока Auto Layout родителя.|

При анализе стилей важно учитывать разделение на `styles` и inline-значения. Figma передает примененные именованные стили в объекте `styles` в виде карты соответствия свойств (например, `fill: style_id`). Сами значения цвета или параметров шрифта при этом дублируются непосредственно в структуре узла (в полях `fills` или `style` для текста). Это позволяет извлекать визуальные параметры без обязательного обращения к таблице стилей, однако для генерации констант темы связывание по идентификаторам стилей остается приоритетным.

Компоненты (`COMPONENT`) и наборы вариантов (`COMPONENT_SET`) передаются через REST API со свойствами в поле `variantProperties` (например, `{"state": "default", "type": "primary"}`). Без использования Plugin API генератор лишен возможности динамически вызывать методы рендеринга или экспорта конкретных инстансов, однако структура вариантов полностью доступна для статического анализа в JSON-ответе.

Variables API (`GET /v1/files/:key/variables/local`) предоставляет прямой доступ к переменным дизайна. По состоянию на 2025–2026 годы этот API имеет критическое ограничение: для его работы требуется подписка класса **Enterprise** и авторизационный токен со scope-доступами `file_variables:read`. Для бесплатных (Free) и начальных (Starter) учетных записей этот эндпоинт возвращает ошибку доступа.

В качестве резервного механизма (fallback) необходимо использовать парсинг глобальной карты `styles` файла и извлечение констант из локальных заливок (`fills`) и текстовых стилей. Дополнительно поле `codeSyntax` в спецификации Variables API позволяет связывать переменные с конкретными строками кода для Dev Mode, что может упростить генерацию токенов при наличии Enterprise-подписки.

Интерфейс Dev Mode / Dev Resources API предоставляет метаданные о связанных с компонентами внешних ссылках и ресурсах. В отличие от плагинов, этот REST API не позволяет генерировать произвольный CSS/Dart код на лету, а лишь отдает статические связи, настроенные проектировщиком вручную.

Экспорт графических ресурсов реализуется через конечную точку `GET /v1/images/:key?ids=...&format=...&scale=...`. Запросы лимитируются по частоте (стандартный Rate Limit — 429 при превышении интенсивности обращений), а возвращаемые временные ссылки на скачивание ресурсов активны в течение 30 дней. Эндпоинт `GET /v1/files/:key/images` служит для извлечения исходных изображений, используемых в качестве текстурных заливок (image fills).

Для интеграции во Flutter применяется следующее дерево принятия решений (Decision Tree): векторные иконки экспортируются в формате SVG и рендерятся через библиотеку `flutter_svg`, в то время как сложные иллюстрации, фотографии и градиенты выгружаются в формате PNG с мультиплексированием масштаба (`@2x`, `@3x`) для предотвращения размытия на экранах с высокой плотностью пикселей.

Официальный репозиторий спецификаций `figma/rest-api-spec` предоставляет актуальное OpenAPI-описание. Написание собственного клиента на базе библиотеки `httpx` и валидационных моделей `Pydantic` в Python предпочтительнее использования сторонних или устаревших SDK, так как это гарантирует полную поддержку новых полей API и легковесность CLI-агента.

### Рекомендуемый подход для MVP

Для минимизации трафика и накладных расходов CLI должен парсить URL фрейма, заменять символы дефиса на двоеточия в ID узла и запрашивать строго целевую ветку через `GET /v1/files/:key/nodes?ids=...`.

Из-за жесткого ограничения Variables API по тарифным планам, в MVP необходимо реализовать гибридный сбор токенов: первичный запрос к Variables API с автоматическим переходом на извлечение токенов из дерева стилей `fills` и `styles` в случае получения HTTP 403. Иконки и растровые картинки должны скачиваться асинхронно параллельными пакетами (batch size до 20 штук) во избежание блокировок по лимиту запросов.

### Риски и ограничения

Основной риск — гарантированная недоступность эндпоинта переменных переменных (`GET /v1/files/:key/variables/local`) у большинства пользователей с базовыми аккаунтами Figma. Разработка MVP исключительно под Variables API сделает продукт нежизнеспособным для широкой аудитории.

Второй лимит — REST API не поддерживает экспорт динамически сгенерированных групп векторов «на лету» без предварительного назначения фрейма экспорта в самом интерфейсе Figma; попытка экспортировать неразмеченные векторные группы может приводить к искажению SVG-файлов.

## Секция 2: Трансформация Figma JSON в чистое дерево дизайна (rule-based слой)

### Результаты исследования (факты и даты)

Исходный JSON, возвращаемый Figma REST API, перегружен метаданными редактора, которые не несут полезной нагрузки для построения интерфейса Flutter. К таким избыточным полям относятся абсолютные координаты (`absoluteBoundingBox`), матрицы трансформации (`relativeTransform`), сложные эффекты наложения слоев, не поддерживаемые базовым Material SDK, а также служебные флаги редактора.

Абсолютное позиционирование полностью противоречит парадигме адаптивной верстки Flutter, поэтому ручное удаление этих параметров снижает объем данных (noise reduction). Исключению подлежат также все невидимые узлы (`visible: false`).

Промежуточный формат данных является ключевым звеном оптимизации контекстного окна LLM. Передача сырого JSON-дерева фрейма среднего размера (около 500 узлов) может расходовать до 150 000 входных токенов. Сжатие структуры данных за счет перевода в плоский или очищенный виджет-ориентированный JSON-формат (схема `CleanDesignTree`) обеспечивает сжатие в пропорции до 15:1.

Сравнительные показатели расхода токенов на один экран:

|**Формат представления данных**|**Средний размер (Кб)**|**Объем в токенах**|**Качество генерации (метрика синтаксических ошибок)**|
|---|---|---|---|
|Сырой Figma JSON|640 KB|~160,000|Высокий риск галлюцинаций и обрыва вывода|
|Очищенное JSON-дерево|38 KB|~9,500|Стабильный синтаксис, точное следование структуре|
|Markdown-спецификация дерева|22 KB|~5,500|Возможна потеря детальных параметров стилей|

Для оптимизации структуры применяется эвристика дедупликации компонентов. Одинаковые визуальные паттерны (например, повторяющиеся элементы списков или кнопки) выявляются по сигнатуре их структуры (количеству дочерних элементов, типам узлов и распределению параметров Auto Layout) либо по текстовым маскам имен слоев (шаблоны вида `Button/Primary`, `Card/Product`). Узлы с типом `INSTANCE` сопоставляются с их родительскими компонентами `COMPONENT`, что позволяет генерировать единый переиспользуемый класс виджета с конструктором вместо дублирования разметки для каждого элемента.

Rule-based предобработка берет на себя рутинные операции по вычислению разметки:

- Преобразование `layoutMode: HORIZONTAL` в `Row`.
    
- Преобразование `layoutMode: VERTICAL` в `Column`.
    
- Автоматическая вставка `Padding` на основе отступов.
    
- Оборачивание дочерних элементов со значением `layoutGrow: 1` в виджеты `Expanded`.
    
- Замена фиксированных размеров на `SizedBox(width:..., height:...)`.
    

При этом за LLM сохраняется интеллектуальная роль принятия семантических решений: выбор между `Card` и `Container`, определение логики скроллинга, замена стандартных кнопок на Material 3 компоненты (`ElevatedButton`, `FilledButton`), а также контекстное связывание визуальных параметров с токенами темы приложения.

Опыт разработки MCP-серверов и CLI-инструментов, таких как `figma-flutter-mcp` и `exfig`, показывает высокую эффективность декларативного описания структуры интерфейса. В частности, проект `figma-spec-mcp` подтверждает жизнеспособность предварительного маппинга сложных узлов разметки перед отправкой в LLM для снижения уровня логических ошибок кодогенерации.

### Риски и ограничения

Если исходный дизайн собран хаотично (без использования механизмов Auto Layout, с нарушением вложенности фреймов), rule-based парсер вернет плоское дерево с преобладанием контейнеров типа `STACK`.

## Секция 3: Генерация кода Flutter Material 3 (2025–2026)

### Результаты исследования (факты и даты)

Важнейшим изменением в экосистеме Flutter стал официальный анонс от 7 апреля 2026 года: **библиотеки Material и Cupertino внутри репозитория `flutter/flutter` заморожены**. Начиная со стабильного релиза Flutter 3.44, развитие визуальных компонентов полностью переносится в независимые пакеты `material_ui` и `cupertino_ui` на pub.dev.

Для разработчиков приложений это означает начало переходного периода: старый встроенный код разметки объявлен устаревшим (deprecated) во всех версиях старше 3.44 и со временем будет полностью удален из ядра. В связи с этим для генератора кода рекомендуется использовать версию SDK не ниже 3.44.0.

YAML

```
environment:
  sdk: ">=3.44.0 <4.0.0"
  flutter: ">=3.44.0"
```

При этом кодогенерация должна строго следовать стандартам Material 3 :

1. Использование конструктора `ColorScheme.fromSeed` для динамического вычисления палитры гармоничных оттенков на основе доминирующего цвета бренда, полученного из токенов Figma.
    
2. Полное исключение устаревших именований текстовых стилей Material 2 (таких как `headline1`, `bodyText1`). Допускаются только актуальные стили M3: `displayLarge`, `headlineMedium`, `titleLarge`, `bodyMedium`, `labelSmall`.
    
3. Адаптивность верстки (mobile-first с возможностью масштабирования под Desktop Web) должна строиться на связке `LayoutBuilder` и `MediaQuery.sizeOf(context)` для предотвращения лишних перерисовок экрана при изменении размеров окна. Стандартные контрольные точки (breakpoints) сообщества: мобильные устройства — до 480px, планшетный режим — до 768px, десктоп — более 1024px.
    
4. Для предотвращения растягивания мобильного интерфейса на всю ширину экрана в Web-демо верстку необходимо оборачивать в центрирующий контейнер `ConstrainedBox` с ограничением `maxWidth: 480`.
    

Автоматизация интеграции ресурсов в `pubspec.yaml` должна выполняться утилитой на Python, гарантирующей безопасность и идемпотентность изменений. Недопустимо использовать регулярные выражения для прямой конкатенации строк, так как это может нарушить форматирование комментариев и структуры YAML. Безопасное слияние должно производиться с помощью структурного парсера (например, библиотеки `ruamel.yaml`), который находит существующий блок `flutter:` и корректно интегрирует зависимости `flutter_svg` и пути к папкам активов `assets/`.

#### Flutter Codegen Style Guide (Правила системного промпта)

1. **Material 3 по умолчанию**: Использовать только виджеты Material 3. Устанавливать `useMaterial3: true` в настройках темы приложения.
    
2. **Типографика M3**: Применять исключительно современные стили из `Theme.of(context).textTheme` (например, `titleMedium`, `bodyLarge`). Никаких жестко заданных имен семейств шрифтов в разметке экрана.
    
3. **Семантика цветов**: Все цвета элементов верстки должны ссылаться на абстрактные поля `ColorScheme` (например, `Theme.of(context).colorScheme.primary`, `onSurface`). Прямое хардкодирование hex-кодов цветов внутри виджетов экрана запрещено.
    
4. **Константные конструкторы**: Автоматически проставлять ключевое слово `const` для всех инициализаций виджетов, не имеющих динамических зависимостей, для снижения нагрузки на сборщик мусора.
    
5. **Ограничение использования Stack**: Виджет `Stack` допускается к генерации только для явных перекрытий элементов и сложной графики, если в clean tree свойство `type` определено как `STACK`. Для последовательного размещения использовать только `Row` и `Column`.
    
6. **Оптимизация списков**: При обнаружении повторяющейся структуры однотипных элементов разметки заменять статический вывод элементов на ленивую загрузку через `ListView.builder` с фиксированной высотой элементов во избежание утечек памяти.
    
7. **Адаптивные отступы**: Избегать фиксированных размеров отступов для мобильных экранов. Размеры отступов и высоты декоративных элементов должны опираться на базовую сетку шагов (`app_spacing.dart`).
    
8. **Безопасные зоны**: Обязательно оборачивать корневой контейнер экрана в виджет `SafeArea` для исключения перекрытия интерфейса системными вырезами экранов мобильных устройств.
    
9. **Обработка векторов**: Все иконки рендерить через `SvgPicture.asset` из библиотеки `flutter_svg`, указывая размеры `width` и `height` в соответствии со значениями `sizing` из clean tree.
    
10. **Растровые изображения**: Фотографии выводить через `Image.asset`, организуя структуру путей с учетом плотности пикселей.
    
11. **Свойство деления файлов**: Большие экраны разделять на приватные подвиджеты внутри одного файла (например, `_HeaderSection`, `_ActionButton`), если размер результирующего Dart-файла превышает 350 строк кода.
    
12. **Доступность**: Добавлять базовое описание элементов управления через виджет `Semantics` (параметры `label` и `hint`) для систем чтения экрана.

## Секция 4: Интеграция LLM для генерации кода Dart (2025–2026)

### Результаты исследования (факты и даты)

По состоянию на 2025–2026 годы технология структурированных ответов (Structured Outputs) в Anthropic Claude API полностью переведена в статус общей доступности (GA). Передача структурированной схемы осуществляется через параметр `output_config.format` со значением типа `"json_schema"`.

Использование старых заголовков и параметров бета-версий (`output_format`) не рекомендуется. Использование структурированного вывода на уровне API накладывает жесткие ограничения на сложность схемы из-за процесса компиляции грамматики декодирования.

Основные технические лимиты Structured Outputs для Claude API:

- **Количество строгих инструментов**: Не более 20 схем с параметром `"strict": true` на один запрос.
    
- **Необязательные параметры**: Суммарно не более 24 параметров, не включенных в массив `required`.
    
- **Параметры с объединением типов**: Не более 16 параметров, использующих `anyOf` или массивы типов (например, `["string", "null"]`) во всех структурах.
    
- **Время компиляции**: При превышении внутренней сложности компиляция схемы может занять более 180 секунд, после чего возвращается ошибка HTTP 400 («Schema is too complex for compilation»).
    
- **Несовместимости**: Использование структурированного вывода полностью несовместимо с функциями цитирования (Citations) и предварительного заполнения ответа (Message Prefilling).
    

Для задачи кодогенерации Flutter-виджетов проведено сравнение эффективности актуального модельного ряда Anthropic:

|**Модель API**|**Стоимость ввода (за 1M)**|**Стоимость вывода (за 1M)**|**Ограничение контекста**|**Оценка качества генерации Dart**|**Рекомендация для MVP**|
|---|---|---|---|---|---|
|**Claude Haiku 4.5**|$1.00|$5.00|200K|Низкая (частые ошибки в сложных вложенных макетах)|Не рекомендуется|
|**Claude Sonnet 4.6**|$3.00|$15.00|1M|Высокая (строгое следование правилам M3, точный JSON)|**Основной выбор**|
|**Claude Opus 4.7**|$5.00|$25.00|1M|Исключительная (глубокий анализ сложных кастомных интерфейсов)|Резерв для сложных экранов|

Claude Sonnet 4.6 обеспечивает оптимальный баланс цены и качества, превосходя модели конкурентов на задачах логического структурирования интерфейсов.

В процессе проектирования кодогенератора рассмотрены три архитектурные альтернативы:

- **Вариант А (Полный вывод файлов)**: LLM возвращает плоский массив объектов `{ "path": str, "content": str }`. Этот подход прост в реализации, но расходует много токенов на бойлерплейт-код и часто приводит к обрывам генерации на больших экранах.
    
- **Вариант В (Структурированный вывод компонентов)**: LLM возвращает структурированный JSON с разделением на классы стилей, атомарные виджеты и тело экрана. Шаблонизатор на Python (Jinja2) собирает файлы на диске. Этот подход является оптимальным для MVP, так как гарантирует чистоту структуры и экономию токенов.
    
- **Вариант С (Генерация AST)**: LLM генерирует абстрактное синтаксическое дерево разметки Dart, которое Python компилирует в код. Архитектура избыточна для MVP из-за высокой сложности поддержки транслятора AST.
    

### Рекомендуемый подход для MVP

Рекомендуется использовать модель **Claude Sonnet 4.6** со структурированным ответом уровня API (`strict: true`) по Варианту В.

## Секция 5: Архитектура Python CLI (2025–2026)

### Рекомендуемый подход для MVP

На основе анализа конкурентов сформирован список лучших практик для заимствования (Steal List).

#### «Steal List» для кодогенератора MVP

1. **Изоляция векторных иконок (из Figma-Flutter-MCP)**: Иконки и иллюстрации, содержащие сложные кривые Безье, не должны разбираться парсером на примитивы Flutter. Их необходимо автоматически помечать в CleanDesignTree как внешние SVG-активы и скачивать асинхронно.
    
2. **Мультиплексирование растра (из ExFig)**: При экспорте растровых картинок (фотографий) CLI должен автоматически скачивать их в нескольких масштабах (`scale=2.0` и `scale=3.0`), размещая файлы в директориях `assets/images/2.0x/` и `assets/images/3.0x/` соответственно, для обеспечения четкости интерфейса на Retina-экранах.
    
3. **Использование единого конфига (из ExFig)**: Заимствование концепции единого декларативного YAML-файла для управления параметрами генерации, путями к папкам и правилами именования файлов, что исключает необходимость ввода длинных цепочек аргументов в консоли.
    
4. **Контракт выравнивания (из figma-spec-mcp)**: На этапе rule-based разбора дерева дизайна автоматически рассчитывать контрастность текста относительно подложки контейнера и выводить предупреждение в CLI, если сгенерированная разметка нарушает рекомендации WCAG AA.

## Секция 7: Безопасность, эксплуатация и подготовка демо-версии

### Результаты исследования (факты и даты)

Безопасность при работе с Figma REST API строится на защите персональных токенов доступа (Personal Access Token, PAT). Токены Figma и ключи Anthropic/OpenAI API предоставляют полный доступ к проектным файлам и балансу учетных записей.

Их утечка через публичные репозитории или файлы логов является критической угрозой безопасности. CLI-агент должен жестко следовать правилам хранения секретов:

- Загрузка секретных ключей должна осуществляться только через переменные окружения или локальный файл `.env`, который принудительно вносится в `.gitignore` при первом запуске утилиты.
    
- В консольном выводе и файлах логов (особенно при запуске с флагом `--verbose`) должен работать автоматический фильтр-маскировщик. Любые строки, похожие на ключи (шаблоны регулярных выражений для токенов Figma `figd_...` и ключей Anthropic `sk-ant-...`), должны заменяться на маску ``.
    
- Запрещено сохранять полный сырой JSON-ответ Figma во временных файлах внутри директории проекта, так как он может содержать конфиденциальные данные проекта или метаданные сотрудников.
    

Для успешной демонстрации работы генератора в MVP-режиме критически важно подготовить эталонную структуру исходного макета в Figma. Поскольку rule-based парсер опирается на Auto Layout, демонстрационный фрейм должен быть спроектирован по строгим правилам:

- Главный контейнер экрана должен иметь статус Frame и включенный Auto Layout (вертикальный или горизонтальный).
    
- Все текстовые блоки должны быть привязаны к стандартным стилям Figma.
    
- Элементы управления (кнопки, поля ввода) должны быть оформлены как Figma-компоненты с явным указанием их состояний в свойствах вариантов.
    
- Иконки должны быть объединены в векторные группы с настроенными параметрами экспорта (Export as SVG).

## Антипаттерны, выявленные в ходе исследования

В процессе анализа технических решений зафиксированы следующие критические антипаттерны, реализация которых в MVP недопустима:

1. **Восприятие Dev Mode API как отдельного «магического» кодогенератора**: Разработчики часто ошибочно полагают, что REST API Dev Mode позволяет напрямую скачивать готовый код виджетов. На самом деле этот API лишь предоставляет доступ к внешним ссылкам (Dev Resources) и статическим свойствам, настроенным дизайнером. Вся кодогенерация интерфейса должна выполняться силами LLM на основе дерева разметки.
    
2. **Передача сырого JSON Figma напрямую в LLM**: Попытка отправить неотфильтрованное дерево JSON из Figma приводит к мгновенному переполнению контекстного окна модели , колоссальным финансовым затратам и резкому росту галлюцинаций LLM из-за обилия мусорных метаданных редактора.
    
3. **Генерация архитектурных библиотек управления состоянием в MVP**: Попытка заставить LLM генерировать сложные связки с `Riverpod`, `Bloc` или настраивать роутинг через `GoRouter` на этапе MVP приводит к генерации неработоспособного связующего кода. MVP должен быть строго сфокусирован на генерации чистой верстки и стилей (Stateless/Stateful виджеты).
    
4. **Прямой вызов Variables API без проверки тарифа пользователя**: Запуск запросов к Variables API без обработки ошибок HTTP 403 гарантированно сломает пайплайн у всех пользователей, не имеющих Enterprise-подписки Figma. Наличие fallback-алгоритма на основе обычных стилей обязательно.
    
5. **Ожидание доступности метода `exportAsync` в REST API**: Разработчики часто путают Plugin API и REST API, ожидая, что через HTTP-запрос можно вызвать динамический рендеринг произвольного узла. REST API умеет экспортировать только те узлы, у которых в самом файле дизайна предварительно настроен блок `exportSettings`.
    
6. **Использование устаревших бета-параметров Claude API**: Применение параметров вида `output_format` или старых бета-заголовков для структурированного вывода приведет к отказу в обслуживании со стороны современных шлюзов Anthropic API в 2026 году. Допускается только актуальный параметр `output_config.format`.
